# 法律角色扮演 RAG 智能问答系统项目讲解文档

## 一、项目概述

本项目是一个面向法律咨询场景的智能问答系统，核心目标是让用户选择不同类型的律师角色后，围绕法律问题进行对话。系统不是单纯调用大模型回答，而是结合本地法律知识库、向量检索、对话记忆和角色提示词，让回答更贴近法律场景。

项目整体可以概括为：

用户在网页端提问，后端接收问题后，根据用户选择的律师角色构造提示词；如果问题需要法律依据，系统会先从 Milvus 向量库中检索相关法律文档片段，再把检索结果、历史对话和用户问题一起交给大模型生成答案，最后把答案、引用依据、风险等级和推荐追问返回给前端。

## 二、项目技术栈

### 1. 后端技术

- FastAPI：提供 HTTP 接口和后端服务。
- Uvicorn：作为 ASGI 服务启动器。
- SQLAlchemy：操作 MySQL 数据库。
- Pydantic：定义请求和响应数据结构。
- Redis：保存短期对话历史。
- Milvus：保存法律文档向量，实现语义检索。
- FlagEmbedding：加载 BGE 嵌入模型和 BGE 重排模型。
- OpenAI SDK：兼容 DeepSeek、OpenAI、Ollama 等大模型接口。

### 2. 前端技术

- HTML、CSS、JavaScript：实现单页前端页面。
- 静态文件由 FastAPI 直接挂载。
- 前端主要功能包括登录注册、选择律师角色、发起提问、流式回答、查看历史会话、上传知识文档。

### 3. 数据库和中间件

- MySQL：保存用户、角色、知识文档元数据。
- Redis：保存用户短期聊天记录。
- Milvus：保存法律知识分块后的 dense 向量和 sparse 向量。
- MinIO、etcd：作为 Milvus standalone 依赖组件。

## 四、核心业务模块

### 1. 用户模块

用户模块位于 `app/api/user.py`，主要负责注册和登录。

用户数据保存在 MySQL 的 `users` 表中，字段包括：

- `id`：用户 ID。
- `username`：用户名。
- `email`：邮箱。
- `hashed_password`：密码。
- `phone`：手机号。
- `is_active`：是否启用。
- `created_at`、`updated_at`：创建和更新时间。

### 2. 角色模块

角色模块位于 `app/api/role.py` 和 `app/core/prompts/lawyer_prompts.py`。

系统支持多个法律角色，例如：

- 综合律师。
- 刑事律师。
- 劳动律师。
- 婚姻家事律师。
- 合同律师。

每个角色都有不同的专业领域和提示词模板。用户选择不同角色后，后端会根据 `role_id` 使用对应的系统提示词，让大模型用不同专业角度回答。

### 3. 知识库模块

知识库模块位于 `app/api/knowledge.py`。

它负责：

- 上传法律文档。
- 解析 PDF、TXT、MD、JSON、CSV、DOCX 等文件。
- 将文档切分成多个文本块。
- 使用 BGE 模型生成向量。
- 把向量和文本内容写入 Milvus。
- 把文档元数据写入 MySQL。

MySQL 中的 `knowledge_documents` 表只保存文档元信息，例如标题、类型、领域、路径、分块数和状态。

真正用于检索的法律内容保存在 Milvus 中。

### 4. 聊天问答模块

聊天接口位于 `app/api/chat.py`，核心业务位于 `app/services/rag_service.py`。

主要接口包括：

- `POST /api/v1/chat`：普通问答。
- `POST /api/v1/chat/stream`：流式问答。
- `GET /api/v1/chat/history/{user_id}`：获取会话历史。
- `DELETE /api/v1/chat/history/{user_id}/{session_id}`：清空指定会话。

聊天模块会完成以下工作：

- 判断问题是否是简单问候或简单说明。
- 获取用户历史对话上下文。
- 根据问题检索相关法律知识。
- 根据角色构建提示词。
- 调用大模型生成回答。
- 整理回答结构。
- 保存用户问题和助手回答。
- 返回引用依据、风险等级和推荐追问。

## 五、数据库设计

### 1. MySQL 数据库

项目使用的 MySQL 数据库名是 `legal_system`。

主要表包括：

```text
users
roles
chat_sessions
knowledge_documents
```

#### users 表

用于保存用户信息。

#### roles 表

用于保存角色信息，例如律师角色的名称、展示名、描述、专业领域、提示词模板。

#### chat_sessions 表

用于保存会话元数据，例如用户 ID、角色 ID、法律领域、会话状态等。

当前项目的短期聊天内容主要保存在 Redis 中。

#### knowledge_documents 表

用于保存知识文档元数据。

它不直接保存向量，只保存：

- 文档标题。
- 文档类型。
- 法律领域。
- 文件路径。
- 分块数量。
- 文档状态。
- 元数据 JSON。

### 2. Redis 数据

Redis 保存短期对话历史，key 形式类似：

```text
legal:chat:{user_id}:{session_id}
```

每条消息包含：

- `role`：user 或 assistant。
- `content`：消息内容。
- `timestamp`：时间。
- `citations`：引用依据。

Redis 的作用是让系统在下一轮对话中知道用户前面说过什么。

### 3. Milvus 向量库

Milvus 集合名是：

```text
legal_knowledge
```

主要字段包括：

- `id`：主键。
- `dense_vector`：稠密向量。
- `sparse_vector`：稀疏向量。
- `content`：法律文本片段。
- `source`：来源。
- `article_number`：条文编号。
- `legal_field`：法律领域。
- `knowledge_type`：知识类型。
- `document_title`：文档标题。

当前集合中已有 366 条向量数据：

```text
中华人民共和国著作权法：90
中华人民共和国婚姻法：11
中华人民共和国民法典：265
```

## 六、RAG 检索流程

RAG 是 Retrieval-Augmented Generation 的缩写，意思是检索增强生成。

本项目的 RAG 流程位于 `app/rag/pipeline.py`。

整体流程如下：

```text
用户问题
  ↓
问题改写
  ↓
BGE 模型生成查询向量
  ↓
Milvus 向量检索
  ↓
可选 rerank 重排
  ↓
整理检索结果
  ↓
拼接到提示词
  ↓
大模型生成答案
```

### 1. 问题改写

系统会根据常见法律关键词扩展问题，例如用户问“离婚”，系统可以扩展出“婚姻关系解除、诉讼离婚、协议离婚”等词，从而提高检索覆盖率。

### 2. 查询向量化

系统使用 BGE 模型把用户问题转换成向量。

之前查询时会分别生成 dense 向量和 sparse 向量，相当于同一个问题跑两次模型。现在已经优化成一次编码，减少了检索前的耗时。

### 3. Milvus 检索

系统默认采用快速检索模式：

- 默认使用 dense 向量检索。
- 默认返回较少候选文档。
- 默认关闭 CPU rerank。
- 默认限制上下文长度。

这样可以显著提升回答速度。

### 4. 重排

重排模型可以提高准确率，但 CPU 环境下速度较慢。

当前默认关闭：

```env
RAG_ENABLE_RERANK=false
```

如果后续机器有 GPU，或者更重视准确率，可以打开。

### 5. 生成回答

检索到的法律片段会和用户问题、历史对话、角色提示词一起传给大模型。

大模型返回后，系统会整理成比较稳定的结构：

```text
核心说明
细分说明
法律依据
```

## 八、一次完整问答的内部执行过程

以用户提问“离婚时夫妻共同财产怎么分”为例：

1. 前端把问题、用户 ID、会话 ID、角色 ID 发送给后端。

2. `app/api/chat.py` 接收请求，调用 `LegalChatService.chat()`。

3. `LegalChatService` 判断这不是简单问候，因此需要走 RAG。

4. 系统从 Redis 中取最近几轮历史对话。

5. `LegalRAGPipeline` 对问题进行改写。

6. BGE 模型把问题转成向量。

7. Milvus 根据向量查找相关法律片段，例如民法典婚姻家庭编相关内容。

8. 系统把检索片段格式化成知识上下文。

9. 系统根据用户选择的律师角色加载对应提示词。

10. 系统将角色提示词、历史对话、法律知识、用户问题拼成完整 prompt。

11. LLMService 调用大模型生成答案。

12. 系统整理答案格式，生成引用依据和推荐追问。

13. 用户问题和助手回答保存到 Redis。

14. 前端展示最终回答。

## 九、项目亮点

### 1. 角色化法律问答

系统不是一个通用问答机器人，而是支持不同律师角色。不同角色的提示词不同，回答角度也不同。

### 2. RAG 知识增强

系统会先检索法律知识，再让大模型回答，降低纯大模型幻觉风险。

### 3. 多数据库协作

项目同时使用 MySQL、Redis、Milvus：

- MySQL 负责结构化业务数据。
- Redis 负责短期对话记忆。
- Milvus 负责法律知识语义检索。

### 4. 支持知识库上传

用户可以上传法律文档，系统自动解析、分块、向量化并入库。

### 5. 支持流式回答

系统提供 `/chat/stream` 接口，可以边生成边展示，用户体验更好。

### 6. 有性能优化设计

当前已经做了以下优化：

- 查询向量化从两次变成一次。
- 默认关闭 CPU rerank。
- 默认使用快速 dense 检索。
- 限制检索上下文长度。
- 限制大模型最大输出长度。
- 设置 1 分钟内必须返回结果。

## 十、可能被问到的问题

### 1. 为什么不用 MySQL 直接查法律内容？

因为用户问题通常不是精确关键词，而是自然语言问题。MySQL 更适合结构化查询，不擅长语义相似度检索。Milvus 可以根据向量相似度找出语义相关的法律片段。

### 2. 为什么要用 Redis？

Redis 读写快，适合保存短期聊天历史。每次问答都需要快速取出最近几轮对话，让模型理解上下文。

### 3. 为什么回答会慢？

主要慢点有两个：

- BGE 向量模型和 rerank 模型在 CPU 上运行会慢。
- 大模型生成本身耗时较长。

项目已经通过快速检索模式和输出长度限制改善速度。

### 4. RAG 和普通大模型问答有什么区别？

普通大模型只依赖模型已有知识。RAG 会先从本地知识库检索相关法律条文，再把这些内容提供给模型，所以答案更可控、更容易引用依据。

### 5. MySQL 和 Milvus 的数据关系是什么？

MySQL 保存文档记录，例如文档标题、状态、分块数。Milvus 保存文档分块后的向量和具体文本片段。MySQL 负责管理，Milvus 负责检索。

## 十一、讲解时可以使用的简短总结

这个项目是一个法律领域的角色扮演 RAG 问答系统。前端提供登录、角色选择、聊天和知识库管理功能；后端使用 FastAPI 提供接口；MySQL 保存用户和文档元数据；Redis 保存短期聊天记忆；Milvus 保存法律文档向量；BGE 模型负责把问题和文档转成向量；大模型负责根据角色提示词和检索到的法律依据生成最终回答。系统的核心流程是“用户提问、检索法律知识、构造提示词、调用大模型、返回结构化答案”。
